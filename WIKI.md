# ideaLab Wiki 维护规范

本文件给维护型 agent 使用。目标是让新材料进入资料库时有稳定流程，而不是不断堆进 `SKILL.md`。

## 一句话原则

`SKILL.md` 是路由，不是资料库。用户说“更新到 skill”时，默认意思是更新这个 ideaLab 知识包。除非要改触发路径、读取顺序或硬边界，否则不要改 `SKILL.md`。

## 维护身份

- 当前主要维护人：用户、Serry/信芮。
- 普通顾问通常只提问，不触发入库。
- 其他同事如果提供新口径或纠错，也可以进入维护流程，但按风险等级处理。
- 无法确认发言人身份时，不靠身份授权，按“内容风险 + 来源可信度 + 是否明确要求更新”判断。

## 两种工作状态

### 答疑状态

用户只是问问题时：

1. 按 `SKILL.md` 路由读取资料。
2. 回答问题。
3. 不改文件，不自动入库。

### 维护状态

用户明确说以下词时进入维护状态：

- 更新、校准、记住、沉淀、入库、清洗、归档、同步资料卡
- 调整字段、优化架构、整理 README、整理 SKILL
- 推送 GitHub、合并、发布包

维护状态必须先判断新内容应该写到哪里。

## “更新到 skill”分流规则

| 内容类型 | 写入位置 | 是否改 `SKILL.md` |

| 校情、学校资料卡、班型、学校索引 | `references/school_knowledge/` | 否 |
| 顾问训练题、模拟家长、评分规则 | `references/consultant_training/` | 否 |
| 回复思路、话术思路、开口方式、追问顺序 | `references/consultant_training/response_playbook.md` | 否 |

## 风险分级

### 低风险，可直接更新

- 错别字、别名、格式、路径修正。
- 已公开官网链接或官方附件的补充。
- 不改变事实的表达优化。
- 已有资料卡的字段补齐。
- Serry/信芮或用户明确给出的低风险校准。

### 中风险，谨慎更新并标来源

- 产品适合人群、推荐顺序、课程关系。
- 校情/赛情判断、学校属性、科技特色、竞争强度。
- 竞品描述。
- 内容生产话术。

处理方式：写入资料卡时必须标明来源、使用边界和置信度。

### 高风险，先入待确认

- 价格、课时、合同、退费。
- 保奖、获奖率、升学、综评、自招、录取。
- 当年赛事截止时间、报名入口、资格条件。
- 校区执行政策、占座、排班、库存。
- 学生、学校、老师、证书、学籍号、手机号、订单、合同等隐私。
- 可能构成商业诋毁或法律风险的竞品表达。

处理方式：先写入 `references/_governance/pending_updates.md`，除非用户或 Serry/信芮明确确认并给出可信来源。

## 入库流程

每次清洗新材料时按这个顺序：

1. 识别材料类型：产品、赛事、区域赛情、校情、顾问训练、竞品、案例、图片/附件。
2. 判断是否含隐私或原始敏感信息。
3. 抽取可复用事实，不保存无必要的原文堆叠。
4. 写入对应资料卡或数据文件。
5. 补齐最小治理字段。
6. 更新对应 README 或索引。
7. 如果新增了全新问题类别，再更新 `SKILL.md` 路由。
8. 如果需要人确认，写入 `pending_updates.md`。
9. 用户明确要求更新时，默认提交并推送 GitHub；除非用户说明先别推、先本地讨论或先给方案。

## 最小治理字段

新增资料卡至少补齐：

- `source_ids` 或 `source_batch`
- `public_safe`
- `risk_level`
- `last_verified`
- `confidence`
- `use_boundary`

如果是结构化 JSON，字段名用英文；如果是 Markdown 资料卡，可以用中文展示，但语义要一致。

## 资料卡字段建议

### 产品卡

保留或补齐：

- 标准名称、别名、当前状态、适用年级/阶段。
- 产品定位、适合谁、不适合谁。
- 核心交付、课时、价格、合同、退费边界。
- 入门条件、相邻产品关系、推荐和避坑提示。
- 主要来源、最近核验、是否可外发。

### 赛事卡

保留或补齐：

- 官方名称、简称/别名、赛事层级、适用学生。
- 报名/申报路径、材料要求、评审阶段、近年时间线。
- 可公开引用数据、产品映射、禁用表达。
- 当年节点必须标来源和核验日期。

### 区域赛事赛情卡

保留或补齐：

- 区域、年份、主要赛事、申报窗口、报名入口。
- 学校推荐机制、材料限制、初评/复评/终评节点。
- 竞争强度、重点学校属性、规划建议。
- 已确认信息和推断信息要分开。

### 学校资料卡

保留或补齐：

- 城市、区、学校名称、学段、学校性质。
- 学校定位、课程/班型特色、科创信号、升学/考试信号。
- 科技特色强度、证据说明、产品适配、顾问追问。
- 使用边界、来源批次、置信度。

`第41届青创赛获奖数` 是学校维度的官方获奖表现数据，可以用于判断该校在第41届青创赛中的科创活跃度和成果表现。不要泛化成学校综合排名或长期趋势。对外海报/文章可写：数据来源：官方获奖数据。

### 顾问训练卡

保留或补齐：

- 场景、家长画像、真实意图、问题。
- 安全回答、追问建议、风险点。
- 评分维度、参考资料、最后核验日期。

### 回复思路和话术思路

适用于 `references/consultant_training/response_playbook.md`。

可以沉淀：

- 这类问题先问什么、后答什么。
- 家长焦虑时怎么承接。
- 价格问题怎么先确认目标再报价。
- 规划问题如何分阶段表达。
- 内部判断如何转换成家长能听懂的话。
- 哪些表达必须降级，哪些话不能说。

不要沉淀：

- 具体价格、课时、退费。
- 赛事当年节点。
- 学校属性、录取信号、学校名单。
- 未脱敏案例或聊天原文。

这些事实应回到对应资料卡。

### 图片和附件

图片可以留在 skill 包里，但要配 manifest，字段包括：

- `asset_id`
- `path`
- `asset_type`
- `source_id`
- `can_send_to_user`
- `usage_context`
- `rights_note`
- `risk_level`
- `last_verified`

没有授权、来源不明或含个人隐私的图片，不标记为可发送。

## 校情和赛情区分

- 校情：城市、学校、班型、科创特色、升学/材料信号，放在 `references/school_knowledge/`。
- 赛情：区级赛事通知、申报窗口、赛事机制、竞争强度，放在 `references/event_knowledge/区域赛事赛情/`。

不要把学生个人成绩、学籍号、证书信息当作“赛情”写入发布包。

## GitHub 规则

- 用户说“更新一下、校准一下、入库、清洗、整理、同步资料卡”等明确维护指令时，完成后默认提交并推送到 GitHub。
- 用户说“先别推、先本地讨论、先看方案、不要更新 GitHub”时，只做本地整理或方案讨论，不提交推送。
- 当前以 `main` 作为唯一稳定主分支。
- 推送前检查是否有原始材料、隐私数据或临时清洗产物被误加入。

## 健康检查

定期检查：

- `SKILL.md` 是否重新变成资料堆。
- 新资料是否有来源、边界和置信度。
- Markdown 资料卡和 JSON 数据是否口径一致。
- 是否存在重复目录或别名目录没有说明。
- 是否有孤立文件没有被 README 或路由索引到。
- 是否有高风险口径没有进入 `pending_updates.md`。
